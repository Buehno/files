Lógica do processo
A avaliação é dividida em duas partes sequenciais: Parte 1 — Base comum (30 min) e Parte 2 — Desafio prático integrado (30 min). Na parte 2, o candidato escolhe uma das três trilhas de profundidade. Isso garante que quem domina apenas uma disciplina ainda mostre excelência nela, enquanto quem domina o conjunto completo é diferenciado pela qualidade da solução integrada.

Escolha da trilha 1 min
Candidato declara em voz alta qual trilha escolherá para a Parte 2. O avaliador registra. Não há penalização pela escolha — o objetivo é mostrar profundidade onde há domínio.

Mini-desafio integrador obrigatório 14 min · incluído nos 40 pts
Candidato recebe um trecho de código Python com um bug + uma query SQL com erro lógico. Deve identificar e corrigir ambos. Não precisa executar — análise estática e explicação em voz alta valem.
## Python — encontre e corrija o bug
def calcular_media(numeros):
    total = 0
    for n in numeros:
        total = total + n
    return total / len(numeros)

resultado = calcular_media([])
print(resultado)

## SQL — encontre o erro lógico
SELECT fornecedor, SUM(valor)
FROM pedidos
WHERE SUM(valor) > 10000
GROUP BY fornecedor;

Parte 2
Enunciado Agente de monitoramento de pedidos 50 pts
Você recebeu acesso a uma API REST simulada de pedidos de compra. Escreva um script Python que:

API mock disponível (usar requests):
GET https://jsonplaceholder.typicode.com/todos?userId=1
(simula lista de pedidos — campo "title" = descrição, "completed" = aprovado)
O script deve:
Buscar os pedidos via requests.get() com tratamento de erro HTTP
Criar uma classe Pedido com atributos id, descricao, aprovado e método resumo() que retorna uma string formatada
Instanciar objetos Pedido a partir do JSON retornado
Filtrar e imprimir apenas os pedidos não aprovados (completed == False)
Salvar o resultado em pendentes.json no diretório local
Bônus (avaliador anota): uso de dataclass, tipagem com typing, ou tratamento de timeout na requisição.
Rubrica de correção
Requisição GET com verificação de status code8 pts
Classe Pedido com atributos corretos e __init__10 pts
Método resumo() funcional com f-string ou format7 pts
Instanciação correta a partir do JSON (loop + construtor)8 pts
Filtro de pendentes correto (not pedido.aprovado)7 pts
Serialização e gravação em pendentes.json10 pts
Total50 pts

Parte 2
Enunciado Mini-agente de triagem de fornecedores com Claude 50 pts
A IAgentics precisa de um script que receba uma lista de fornecedores em texto livre e use um LLM para classificá-los automaticamente. O candidato usará a API da Anthropic (key fornecida pelo avaliador durante a prova).
## Dados de entrada (fornecidos como constante no script)
FORNECEDORES = [
    "Empresa Alpha - entrega em 48h, prazo 30d, score 4.2",
    "Beta Suprimentos - atrasos frequentes, prazo 15d, score 2.1",
    "Gamma Tech - certificada ISO, prazo 45d, score 4.8",
]
O script deve:
Montar um system prompt que instrua o modelo a classificar cada fornecedor como PRIORITÁRIO, REGULAR ou RISCO
Chamar anthropic.messages.create() (SDK Python) passando a lista como user message
Parsear a resposta do modelo e imprimir cada fornecedor com sua classificação
Explicar em comentários no código por que cada decisão de prompt foi tomada
Bônus: uso de structured output via JSON mode, tratamento de rate limit com retry, ou prompt com chain-of-thought explicitado.
Rubrica de correção
System prompt com instrução clara de classificação e categorias definidas12 pts
Chamada correta à API Anthropic com parâmetros válidos (model, max_tokens, messages)10 pts
Extração e exibição correta do conteúdo da resposta (.content[0].text)8 pts
Comentários no código explicando intenção do prompt (raciocínio demonstrado)10 pts
Saída legível associando cada fornecedor à sua classificação10 pts
Total50 pts

Parte 2
Parte A — SQL Análise de pedidos de compra 35 pts
O avaliador fornece um arquivo pedidos.db SQLite com as tabelas abaixo. O candidato deve abrir com sqlite3 via terminal ou DB Browser e executar as queries.
-- Schema fornecido
CREATE TABLE pedidos (
  id INTEGER PRIMARY KEY,
  fornecedor_id INTEGER,
  valor REAL,
  status TEXT,       -- 'aprovado' | 'pendente' | 'cancelado'
  criado_em TEXT     -- formato 'YYYY-MM-DD'
);
CREATE TABLE fornecedores (
  id INTEGER PRIMARY KEY,
  nome TEXT,
  categoria TEXT     -- 'insumo' | 'servico' | 'equipamento'
);

-- Dados de exemplo já inseridos (20 registros)
Escreva as queries:
Total gasto por categoria de fornecedor, somente pedidos aprovados, ordenado decrescente 10 pts
Nome dos fornecedores que têm mais de 2 pedidos pendentes 12 pts
Média de valor por mês em 2024 (extrair mês de criado_em) 13 pts
Parte B — Git Workflow de contribuição 15 pts
No terminal, com um repositório local já inicializado e com um commit inicial, o candidato deve executar ao vivo:
Criar branch feature/analise-pedidos e mudar para ela 3 pts
Criar um arquivo queries.sql, adicionar ao stage e commitar com mensagem semântica 5 pts
Voltar para main e fazer merge da feature branch 4 pts
Usar git log --oneline e explicar o histórico exibido 3 pts
